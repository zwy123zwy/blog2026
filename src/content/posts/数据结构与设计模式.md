# 数据结构与设计模式

本文在本地整理**常见数据结构**与**设计原则、全部设计模式**，便于面试复习与日常查阅。结构：一、数据结构；二、设计原则（SOLID 等）；三、全部设计模式（23 种 GoF，含设计思想与详解）；四、面试要点速查。

---

## 一、数据结构

### 1. 数组（Array）

- **特点**：连续存储、下标 O(1) 访问、插入/删除中间元素需移动，O(n)。
- **常见操作**：遍历、双指针、前缀和、二分查找、滑动窗口。
- **时间复杂度**：访问 O(1)、末尾增删 O(1)（摊还）、中间增删 O(n)、查找未排序 O(n)、二分 O(log n)。

### 2. 链表（Linked List）

- **特点**：结点通过指针串联，不要求连续存储；单链表、双向链表、循环链表。
- **常见操作**：虚拟头结点、快慢指针、反转、合并、找中点、判环。
- **时间复杂度**：按下标访问 O(n)、头尾增删 O(1)（已知结点时）、查找 O(n)。

### 3. 栈（Stack）

- **特点**：后进先出（LIFO），仅在一端进行 push/pop。
- **常见操作**：括号匹配、表达式求值、单调栈（下一个更大/更小元素）、DFS 非递归、撤销/历史。
- **时间复杂度**：push / pop / peek 均为 O(1)。

### 4. 队列（Queue）

- **特点**：先进先出（FIFO），一端入队、另一端出队。
- **常见操作**：BFS、滑动窗口最值（双端队列）、任务调度、缓冲。
- **变体**：双端队列（Deque）、优先队列（堆实现）、循环队列。
- **时间复杂度**：入队 / 出队 O(1)（用链表或循环数组实现）。

### 5. 哈希表（Hash Table）

- **特点**：键值存储，通过哈希函数将 key 映射到桶，理想情况下查找、插入、删除 O(1)。
- **常见操作**：计数、去重、两数之和、分组、缓存（LRU 常配合双向链表）。
- **注意**：哈希冲突（链地址法、开放寻址）、负载因子、不可变对象作 key（如 JS 中对象引用作 key）。

### 6. 树（Tree）

- **二叉树**：每个结点最多两子；二叉搜索树（BST）左小右大，中序遍历有序；平衡树（AVL、红黑树）保证高度 O(log n)。
- **遍历**：前/中/后序（递归或栈）、层序（队列）。
- **常见题**：最大深度、路径和、最近公共祖先（LCA）、序列化/反序列化、BST 验证与恢复。
- **堆（Heap）**：完全二叉树，父结点与子结点满足大小关系；大顶堆/小顶堆；优先队列、Top K、合并 K 路有序链表。

### 7. 图（Graph）

- **存储**：邻接表、邻接矩阵；有向/无向、带权/无权。
- **遍历**：DFS（栈/递归）、BFS（队列）。
- **常见算法**：最短路径（BFS 无权、Dijkstra、Bellman-Ford）、拓扑排序、并查集（连通分量）、最小生成树（Prim、Kruskal）。

### 8. 复杂度速查

| 结构     | 访问     | 查找     | 插入     | 删除     |
|----------|----------|----------|----------|----------|
| 数组     | O(1)     | O(n)/O(log n) 二分 | O(n) 中间 | O(n) 中间 |
| 链表     | O(n)     | O(n)     | O(1) 已知结点 | O(1) 已知结点 |
| 栈/队列  | —        | —        | O(1)     | O(1)     |
| 哈希表   | —        | O(1) 均摊 | O(1) 均摊 | O(1) 均摊 |
| BST      | O(log n) 理想 | O(log n) | O(log n) | O(log n) |
| 堆       | O(1) 堆顶 | O(n)     | O(log n) | O(log n) 堆顶 |

## 二、设计原则

设计原则是指导**如何抽象、如何划分职责、如何依赖**的指导思想；设计模式是在具体场景下对这些原则的运用。面试常考 SOLID 与常见口诀。

### 1. SOLID 五原则

| 原则 | 英文 | 设计思想 | 要点 |
|------|------|----------|------|
| **单一职责** | Single Responsibility (SRP) | 一个类/模块只负责一类变化、一种职责。 | 改动的理由只有一个；避免“上帝类”；便于测试与复用。 |
| **开闭** | Open-Closed (OCP) | 对扩展开放，对修改关闭。 | 通过新增代码（子类、策略、插件）扩展行为，而非改原有实现。 |
| **里氏替换** | Liskov Substitution (LSP) | 子类应能替换父类使用，不破坏调用方预期。 | 子类不缩小前置条件、不放大后置条件；继承表示“是一个”且行为可替换。 |
| **接口隔离** | Interface Segregation (ISP) | 客户端不应依赖用不到的接口。 | 接口细粒度、按需组合，避免“胖接口”导致实现类被迫实现无用方法。 |
| **依赖倒置** | Dependency Inversion (DIP) | 依赖抽象而非具体实现；高层模块不直接依赖低层。 | 面向接口/抽象编程；通过注入（构造、setter、容器）解耦。 |

**记忆**：**S** 单一类一责、**O** 开扩关改、**L** 子可替父、**I** 接口要小、**D** 依赖抽象。

### 2. 其他常用原则

- **DRY（Don't Repeat Yourself）**：不重复自己。相同逻辑抽成函数/模块/配置，避免多处复制粘贴，改一处即全局生效。
- **KISS（Keep It Simple, Stupid）**：保持简单。能简单实现就不过度设计；先满足当前需求，再考虑扩展。
- **LoD / 迪米特法则（Law of Demeter）**：最少知识。对象只与直接朋友通信，少依赖间接依赖；减少耦合、避免链式调用暴露内部结构。
- **合成复用**：优先用组合（has-a）而非继承（is-a）复用行为。组合更灵活、避免继承层次过深与脆弱的基类修改。

### 3. 原则与模式的关系

- **单一职责** → 拆类、拆模块；**策略、状态**等把“一类行为”独立成策略/状态类。
- **开闭** → **策略、装饰器、模板方法、观察者**等通过扩展新类而非改原类。
- **依赖倒置** → **工厂、依赖注入、抽象工厂**等让高层依赖抽象。
- **接口隔离** → **适配器、外观**等可提供小接口；**桥接**分离抽象与实现。

## 三、设计模式（全部 23 种 GoF）

GoF（Gang of Four）将 23 种模式分为**创建型、结构型、行为型**。下面按类列出，每种含**设计思想**与**详解**。

### （一）创建型（5 种）

关注**对象的创建**，将创建与使用解耦，避免硬编码 new 与复杂构造逻辑。

---

#### 1. 单例模式（Singleton）

- **设计思想**：全局唯一实例，集中管理共享资源或状态，避免多份导致不一致或浪费。
- **详解**：构造私有化，对外提供唯一访问点（如 `getInstance()`）；注意多线程下的懒汉需双重检查锁或静态内部类；JS 中模块导出对象即天然单例。
- **适用**：配置、连接池、日志、全局 Store、线程池。
- **示例**：`const config = { ... }; export default config;` 或 class 内静态实例 + 私有 constructor。

#### 2. 工厂方法（Factory Method）

- **设计思想**：将“创建具体产品”推迟到子类，父类只定义创建接口；符合开闭，新增产品只需新增子类。
- **详解**：Creator 中有抽象方法 `factoryMethod()` 返回 Product；ConcreteCreator 实现该方法 new 出 ConcreteProduct。调用方依赖抽象 Product，不依赖具体类。
- **适用**：框架提供扩展点（如插件创建）、多态创建、避免 if/switch 堆砌。
- **示例**：`createButton()` 在 WinFactory 中 return new WinButton()，在 MacFactory 中 return new MacButton()。

#### 3. 抽象工厂（Abstract Factory）

- **设计思想**：创建**一族**相关或依赖的产品，保证这些产品一起使用时的兼容性（如同一主题的按钮+输入框）。
- **详解**：抽象工厂接口声明多个创建方法（如 createButton、createInput）；每个具体工厂实现一族具体产品。客户端只依赖抽象工厂与抽象产品。
- **适用**：UI 主题、跨平台控件、数据库+报表一族。
- **与工厂方法区别**：工厂方法是一个方法创建一个产品；抽象工厂是一组方法创建一族产品。

#### 4. 建造者（Builder）

- **设计思想**：把复杂对象的**构建与表示**分离，分步设置参数，最后统一组装；避免构造参数过多、可选参数组合爆炸。
- **详解**：Director 按步骤调用 Builder 的 setX、setY；ConcreteBuilder 内部累积状态，在 build() 中返回最终对象。链式调用（return this）常见于 JS。
- **适用**：配置对象、SQL 构建、链式 API、多参数且多可选的 DTO。
- **示例**：`query.select('a').where('id', 1).orderBy('time').build()`。

#### 5. 原型（Prototype）

- **设计思想**：通过**复制已有对象**创建新对象，而不是 new + 逐字段赋值；适合创建成本高或需隔离初始状态。
- **详解**：原型类实现 clone()（浅拷贝或深拷贝）；客户端调用 clone 得到新实例。JS 中 Object.create、展开运算、结构化克隆可视为原型思想。
- **适用**：初始化代价大、需要对象副本、配合工厂从模板克隆。

### （二）结构型（7 种）

关注**类与对象的组合**，得到更大结构的同时保持灵活与可复用。

---

#### 6. 适配器（Adapter）

- **设计思想**：将**不兼容的接口**转成客户端期望的接口，使已有类可被复用而不改其源码。
- **详解**：Adapter 实现目标接口，内部持有被适配者（Adaptee），在目标方法中调用 Adaptee 并做参数/返回值转换。类适配器用继承，对象适配器用组合（更常用）。
- **适用**：旧接口兼容、第三方库封装、统一不同数据源为同一接口。
- **示例**：将 legacyApi.getData(cb) 适配成 Promise 或 async/await。

#### 7. 桥接（Bridge）

- **设计思想**：将**抽象与实现**分离，使二者可独立变化；用组合代替多层继承。
- **详解**：Abstraction 持有 Implementor；RefinedAbstraction 扩展抽象，ConcreteImplementor 实现细节。例如“形状”+“绘制实现”（像素/矢量）可独立扩展。
- **适用**：多维度变化（如 UI 控件 × 主题）、避免类爆炸、平台相关实现与业务逻辑解耦。

#### 8. 组合（Composite）

- **设计思想**：将对象组织成**树形结构**，使客户端对单个对象与组合对象使用一致（统一当成“组件”）。
- **详解**：Component 定义 add/remove/getChild 及 operation()；Leaf 实现 operation；Composite 持有子 Component 列表，operation 时遍历子节点。树形 UI、文件系统、表达式树常用。
- **适用**：树形结构、部分-整体一致对待、菜单/目录/组织架构。

#### 9. 装饰器（Decorator）

- **设计思想**：**动态**给对象增加职责，比继承更灵活；不改原类，通过包装一层（或多层）增加行为。
- **详解**：Decorator 与 Component 同接口，内部持有 Component，在方法中先调用原方法再执行增强逻辑。可多层嵌套。JS 中 HOC、中间件、withLog 均为装饰思想。
- **适用**：日志、权限、缓存、重试、参数校验、前端 HOC。
- **示例**：`withLog(withAuth(fn))`。

#### 10. 外观（Facade）

- **设计思想**：为子系统提供**统一、简化的入口**，隐藏内部多类、多步骤，降低客户端与子系统的耦合。
- **详解**：Facade 提供少量高层方法，内部调用多个子模块完成复杂流程。客户端只依赖 Facade。
- **适用**：封装 SDK、简化 API、兼容旧接口、分层时上层对下层的门面。

#### 11. 享元（Flyweight）

- **设计思想**：用**共享**减少大量细粒度对象的内存；把可共享的状态（内在）与不可共享的状态（外在）分离，外在由调用时传入。
- **详解**：FlyweightFactory 缓存 Flyweight；Flyweight 只存内在状态；使用时把外在状态作为参数传入方法。例如字符、棋子、连接池。
- **适用**：大量相似对象、对象可区分内在/外在状态、内存敏感场景。

#### 12. 代理（Proxy）

- **设计思想**：为对象提供**代理**，控制访问、延迟加载、权限、日志等，而不改原对象。
- **详解**：Proxy 与 RealSubject 同接口，内部持有一个 RealSubject（或按需创建）；在方法前后加逻辑。ES6 Proxy 可拦截 get/set 等，Vue 响应式、校验、埋点常用。
- **适用**：远程代理、虚拟代理（懒加载）、保护代理、缓存代理、ES6 Proxy 做响应式。

### （三）行为型（11 种）

关注**对象间的职责分配与通信**，使行为可扩展、可复用、可编排。

---

#### 13. 责任链（Chain of Responsibility）

- **设计思想**：请求沿**链**传递，直到有对象处理；发送者与处理者解耦，可动态调整链。
- **详解**：Handler 有 successor，handleRequest() 中若自己能处理则处理，否则交给 successor。中间件、审批流、事件冒泡均为责任链思想。
- **适用**：多级审批、中间件管道、异常/错误处理链、事件传递。

#### 14. 命令（Command）

- **设计思想**：将**请求封装成对象**，便于参数化、排队、日志、撤销/重做。
- **详解**：Command 接口有 execute()（及 undo()）；ConcreteCommand 持有 Receiver 与参数，execute 时调用 Receiver 的方法。Invoker 只持有一个 Command 并调用 execute。
- **适用**：撤销/重做、任务队列、宏命令、异步命令封装。

#### 15. 解释器（Interpreter）

- **设计思想**：为**某种语言/表达式**定义文法并解释执行；将句子表示成抽象语法树，各节点解释自身。
- **详解**：Context 保存输入与状态；Expression 有 interpret(context)；终结符与非终结符表达式递归解释。适合简单文法，复杂文法用解析器生成器。
- **适用**：简单 DSL、表达式求值、规则引擎、正则等。

#### 16. 迭代器（Iterator）

- **设计思想**：**统一遍历**集合的方式，不暴露内部表示；支持多种遍历顺序而不改集合类。
- **详解**：Iterator 提供 next()、hasNext() 等；Aggregate 提供 createIterator()。JS 中 Symbol.iterator、for...of 即迭代器协议。
- **适用**：统一遍历、惰性遍历、隐藏集合结构、多种遍历策略。

#### 17. 中介者（Mediator）

- **设计思想**：用**中介对象**封装对象间的交互，使多对多变为一对多，降低耦合。
- **详解**：Colleague 之间不直接引用，只与 Mediator 通信；Mediator 协调各 Colleague。例如聊天室、表单校验协调器、弹窗与按钮的联动。
- **适用**：对象间交互复杂、希望集中管理交互、避免网状引用。

#### 18. 备忘录（Memento）

- **设计思想**：在不破坏封装的前提下**保存与恢复**对象内部状态，用于撤销、快照、持久化。
- **详解**：Originator 产生状态，通过 createMemento() 交给 Memento 保存；Caretaker 持有 Memento，需要时 restore(memento) 恢复。注意 Memento 只暴露给 Caretaker。
- **适用**：撤销、快照、保存/加载、状态回放。

#### 19. 观察者（Observer）

- **设计思想**：定义**一对多**依赖，主题状态变化时自动通知所有订阅者，使对象松耦合。
- **详解**：Subject 维护 observers 列表，有 attach/detach/notify；Observer 有 update(data)。发布订阅是观察者的变体，通过事件通道进一步解耦发布者与订阅者。
- **适用**：事件系统、数据绑定、UI 与数据解耦、消息推送。
- **示例**：Vue 的响应式、EventEmitter、DOM 事件。

#### 20. 状态（State）

- **设计思想**：将**对象行为随状态变化**的逻辑拆成多个状态类，状态切换时行为随之改变，避免巨大 switch/if。
- **详解**：Context 持有当前 State；State 接口有 handle()；ConcreteState 实现不同行为并在适当时将 Context 的 state 设为下一状态。与策略类似，但状态会主动切换。
- **适用**：状态机、工作流、订单状态、连接状态（如 TCP）。

#### 21. 策略（Strategy）

- **设计思想**：将**算法/规则**封装成可互换的策略，上下文依赖策略接口，运行时注入不同策略。
- **详解**：Context 持有 Strategy；Strategy 接口有 execute(data)；ConcreteStrategy 实现不同算法。调用方通过 setStrategy 或构造注入切换策略。
- **适用**：排序、校验、计价、表单提交策略、多套规则可切换。
- **与状态区别**：策略由外部指定，一般不自动切换；状态在内部根据条件切换。

#### 22. 模板方法（Template Method）

- **设计思想**：在父类中定义**算法骨架**，把部分步骤推迟到子类实现；子类只填“空”，不改变调用顺序。
- **详解**：AbstractClass 有 templateMethod() 按顺序调用 step1()、step2() 等；部分 step 为抽象方法，由子类实现。钩子方法可选，子类可覆盖以影响流程。
- **适用**：框架生命周期、固定流程（如读配置→校验→执行→写日志）、复用流程骨架。

#### 23. 访问者（Visitor）

- **设计思想**：将**作用于元素上的操作**从元素类中分离出来，封装到 Visitor 中；便于新增操作而不改元素类，但新增元素需改 Visitor。
- **详解**：Element 有 accept(visitor)；Visitor 为每种 ConcreteElement 定义 visitX(element)。双分派：元素接受访问者，访问者再根据元素类型执行对应 visit。适合元素稳定、操作常增的场景。
- **适用**：AST 遍历与处理、编译器、文档导出（多格式）、复杂结构上的多种操作。

### 设计模式总览表

| 类型 | 模式 | 设计思想一句话 |
|------|------|----------------|
| 创建型 | 单例 | 全局唯一实例，集中管理。 |
| 创建型 | 工厂方法 | 创建推迟到子类，扩展新产品不改原类。 |
| 创建型 | 抽象工厂 | 创建一族相关产品，保证兼容。 |
| 创建型 | 建造者 | 分步构建复杂对象，构建与表示分离。 |
| 创建型 | 原型 | 通过复制已有对象创建新对象。 |
| 结构型 | 适配器 | 将不兼容接口转为目标接口。 |
| 结构型 | 桥接 | 抽象与实现分离，独立变化。 |
| 结构型 | 组合 | 树形结构，对单体和组合一致对待。 |
| 结构型 | 装饰器 | 动态包装，增加职责不改原类。 |
| 结构型 | 外观 | 为子系统提供统一简化入口。 |
| 结构型 | 享元 | 共享细粒度对象，分离内外状态。 |
| 结构型 | 代理 | 为对象提供代理，控制访问与增强。 |
| 行为型 | 责任链 | 请求沿链传递直至被处理。 |
| 行为型 | 命令 | 请求封装成对象，支持排队与撤销。 |
| 行为型 | 解释器 | 定义文法并解释执行。 |
| 行为型 | 迭代器 | 统一遍历集合，不暴露内部结构。 |
| 行为型 | 中介者 | 封装多对象交互，降低耦合。 |
| 行为型 | 备忘录 | 保存与恢复对象状态。 |
| 行为型 | 观察者 | 一对多依赖，状态变化自动通知。 |
| 行为型 | 状态 | 行为随状态变化，状态类封装。 |
| 行为型 | 策略 | 算法可互换，运行时注入。 |
| 行为型 | 模板方法 | 骨架固定，步骤子类实现。 |
| 行为型 | 访问者 | 操作与元素分离，操作集中到访问者。 |

## 三（续）、设计模式在前端开发中的应用

下面按**模式**或**场景**说明设计模式在**前端、Vue、React**中的典型应用，便于在项目与面试中对应。

### 1. 通用前端场景（与框架无关）

| 模式 | 在前端中的典型应用 |
|------|--------------------|
| **单例** | 全局 Router、Redux/Vuex/Pinia Store、请求实例（axios 单例）、全局 EventBus、配置对象、SDK 初始化（如埋点、地图） |
| **工厂 / 抽象工厂** | 根据 type/config 创建不同组件（如 FormItem 根据 fieldType 渲染 Input/Select）、创建不同请求适配器（fetch/xhr）、多主题/多端 UI 组件族 |
| **建造者** | 链式 API：axios 配置（`.baseURL().timeout().interceptors`）、QueryBuilder、表单/表格配置对象的分步构建、Webpack/Vite 配置 |
| **原型** | 复制表单初始值、深拷贝配置、从模板对象克隆新实例（如默认 props 拷贝） |
| **适配器** | 将后端 REST 数据适配成表格/图表所需格式、将回调式 API 包装成 Promise、旧版 API 兼容新版、不同 UI 库组件接口统一 |
| **外观** | 封装复杂第三方 SDK（地图、支付、IM）为简单 API、封装「请求 + 缓存 + 错误处理」为 useRequest、BFF 层对多接口聚合 |
| **装饰器** | 高阶函数：withAuth、withLog、memo、日志/埋点/权限包装组件或方法；中间件（Koa/Express、Redux middleware） |
| **代理** | ES6 Proxy：Vue 3 响应式、表单校验拦截、读写埋点；图片懒加载（代理 img src）；API 请求代理与缓存 |
| **观察者 / 发布订阅** | DOM 事件、EventEmitter、自定义事件总线、数据绑定与响应式更新、WebSocket 消息分发 |
| **策略** | 表单校验规则（必填/邮箱/手机可切换）、支付方式（支付宝/微信）、排序/筛选策略、路由守卫策略、国际化文案策略 |
| **状态** | 订单状态流转、连接状态（idle/loading/error）、步骤条、多 Tab 多状态 UI、播放器状态（播放/暂停/缓冲） |
| **责任链** | 中间件：Koa/Express 中间件、Axios 拦截器、Vue Router 导航守卫、请求/响应管道、错误处理链 |
| **命令** | 撤销/重做（富文本、画布）、操作队列、宏命令（批量执行）、异步命令封装（Promise/async） |
| **模板方法** | 生命周期钩子（beforeCreate → created → mounted…）、请求封装（beforeRequest → request → afterResponse）、页面/组件骨架与可覆盖步骤 |
| **组合** | 树形组件（菜单、目录、组织架构）、递归组件、Slot 与嵌套结构、虚拟 DOM 树、配置驱动的表单/表格 |
| **迭代器** | for...of、Generator、自定义列表/树遍历、分页/懒加载迭代、Symbol.iterator 与可迭代协议 |
| **中介者** | 复杂表单多字段联动、弹窗与按钮/表单的协调、多组件通信集中到 Store 或 EventBus、聊天室/协作场景 |
| **备忘录** | 路由/滚动位置恢复、草稿保存、撤销栈、快照与时间旅行调试（Redux DevTools）、localStorage 持久化状态 |

### 2. Vue 中的体现

| 模式 | 在 Vue 中的具体体现 |
|------|---------------------|
| **代理** | **Vue 3 响应式**：`reactive()` 用 `Proxy` 代理对象，在 get/set 中做依赖收集与触发更新；Vue 2 用 `Object.defineProperty` 也是“访问代理”思想。 |
| **观察者** | **依赖收集与 Watcher**：data 变化 → Dep 通知 Watcher → 更新视图；`$emit` / 事件总线是发布订阅；Vue 3 的 effect + track/trigger 同样是观察者思想。 |
| **组合** | **组件树**：父/子/递归组件；模板中的嵌套标签与 slot；虚拟 DOM 树。 |
| **发布订阅** | `$on` / `$off` / `$emit`；Vue 3 移除实例事件后，可用 mitt 等库；跨组件非父子通信。 |
| **策略** | 不同输入类型的校验规则、不同表单控件的渲染策略；自定义指令的不同逻辑。 |
| **适配器** | `v-model` 对不同表单元素的适配（input 用 value/input，checkbox 用 checked/change）；将后端字段名映射为前端展示。 |
| **外观** | `createApp()` 与 `use(plugin)` 封装了运行时、路由、状态等；Composition API 的 `useXxx` 封装复杂逻辑为简单入口。 |
| **单例** | Vue 应用实例（一个应用一个 `createApp()`）；Pinia/Vuex 的 store 在应用内单例；全局 router。 |
| **依赖倒置** | `provide` / `inject`：子组件依赖“抽象”（inject 的 key），不依赖具体父组件实现；插件与依赖注入。 |
| **模板方法** | 生命周期钩子（beforeCreate → created → … → unmounted）；可视为“框架定义骨架，用户实现步骤”。 |
| **状态** | 组件内部 loading/success/error；步骤条、多 Tab；用组合式函数封装状态机（如 useStep）。 |

### 3. React 中的体现

| 模式 | 在 React 中的具体体现 |
|------|------------------------|
| **组合** | **组件树与 children**：组件嵌套、`props.children`、复合组件（如 Select + Option）；Fiber 树与虚拟 DOM。 |
| **装饰器 / 高阶组件 HOC** | `withRouter`、`connect`（Redux）、`memo`、自定义 HOC（withAuth、withLog）；包装组件以增加行为而不改原组件。 |
| **观察者 / 订阅** | **状态更新与重渲染**：setState/useState 变化 → 订阅的组件重渲染；Context 的 Provider 变更通知 Consumer；发布订阅库（如 EventEmitter）。 |
| **策略** | 按 props 切换渲染逻辑（如 `type="primary"` 与 `type="link"`）；Render Props 传入“渲染策略”；自定义 Hooks 传入策略函数。 |
| **状态** | **useState + 条件分支** 表示多状态；useReducer 管理复杂状态机；状态机库（XState）与 State 模式一致。 |
| **单例** | React 应用根节点、全局 Context.Provider、Redux store、Router 实例。 |
| **工厂** | 根据 `type` 或 `component` 动态创建组件（如 `<Form.Item component={typeMap[field.type]} />`）；React.createElement。 |
| **命令** | 撤销/重做（useReducer + 历史栈）、操作队列、setState 的更新函数形式（函数式更新）。 |
| **责任链** | 中间件：Redux middleware、自定义 hooks 管道、错误边界层级处理。 |
| **模板方法** | 类组件的生命周期（componentDidMount 等）；函数组件中“固定流程 + 自定义逻辑”的 hooks（如 useEffect 依赖与清理）。 |
| **外观** | `createRoot()`、`Provider` 组合；自定义 Hook 封装“请求 + 缓存 + 错误”（如 useRequest）为简单 API。 |
| **代理** | 不直接体现为 Proxy，但“容器组件 + 展示组件”中容器负责数据与代理式转发；部分状态库内部用 Proxy 做不可变检测。 |
| **依赖倒置** | Context：组件消费抽象“数据来源”，不依赖具体 Provider 实现；依赖注入与测试时替换 Provider。 |
| **迭代器** | 列表渲染（map）、Generator、懒加载列表（虚拟列表迭代渲染）。 |

### 4. 面试可答要点

- **Vue**：响应式 = **代理**（Proxy/defineProperty）+ **观察者**（Dep/Watcher 或 effect）；组件树 = **组合**；生命周期 = **模板方法**；provide/inject = **依赖倒置**；v-model 对不同控件 = **适配器**。
- **React**：组件组合 = **组合**；HOC / 包装组件 = **装饰器**；状态更新与重渲染 = **观察者**；Context = **依赖倒置** 与 部分 **单例**；useReducer/状态机 = **状态**；Render Props / 传入渲染函数 = **策略**。
- **通用**：Router/Store 单例、请求/校验策略、中间件责任链、撤销重做命令、表单/表格组合与状态，是面试中常举的前端设计模式例子。

## 四、面试要点速查

- **设计原则**：能说出 SOLID 五条及含义（单一类一责、开扩关改、里氏替换、接口隔离、依赖倒置）；能提 DRY、KISS、迪米特、合成复用。
- **设计模式**：能按创建型/结构型/行为型各举 2～3 个，说明**设计思想**与**适用场景**；常考单例、工厂、观察者、策略、装饰器、代理、模板方法、责任链；能区分策略与状态、工厂方法与抽象工厂、装饰器与代理；能结合 **Vue/React** 说明各模式在框架中的体现（如 Vue 响应式=代理+观察者、React HOC=装饰器、Context=依赖倒置）。
- **数据结构**：各结构特点与复杂度、典型题思路（链表、二叉树、哈希、栈、堆）。
- **综合**：在业务中何时引入哪种模式（扩展不改原类→开闭与策略/装饰器；统一创建→工厂；解耦通知→观察者；统一遍历→迭代器）；原则与模式的对应关系。
