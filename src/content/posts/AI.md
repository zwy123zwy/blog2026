# AI 应用领域面试知识框架

## 一、AI/ML 基础

### 1.1 机器学习基本概念

- **监督学习 / 无监督学习 / 强化学习**：区别、典型任务与算法。
- **过拟合与正则化**：L1/L2、Dropout、早停、数据增强。
- **偏差与方差**：欠拟合 vs 过拟合，偏差-方差权衡。
- **评估指标**：准确率、精确率、召回率、F1、AUC-ROC、困惑度（Perplexity）。
- **交叉验证**：K-Fold、留一法，在 LLM 场景下如何做评估。

### 1.2 深度学习基础

- **反向传播与梯度下降**：SGD、Adam、学习率调度。
- **常见结构**：CNN、RNN/LSTM、Transformer 的适用场景。
- **注意力机制**：Self-Attention、多头注意力，与 Transformer 的关系。
- **预训练与微调**：预训练目标（MLM、CLM）、微调方式（全量、LoRA、Adapter）。

## AI 基础概念



### 人工智能（AI）



人工智能（Artificial Intelligence）是让计算机模拟人类智能的技术。简单来说，就是让机器能像人一样思考、学习和解决问题。

在 Vibe Coding 中，AI 就是你的编程助手。你告诉它要做什么，它帮你写代码。就像你有一个 24 小时在线的程序员朋友，随时可以帮你干活。

### 大语言模型（LLM）



大语言模型（Large Language Model）是一种能够理解和生成人类语言的 AI 系统。ChatGPT、Claude、Gemini、DeepSeek 都是大语言模型。

为什么叫 “大” 呢？因为这些模型的参数量非常庞大，动辄几十亿甚至上万亿个参数。参数越多，模型通常越聪明，但也越消耗计算资源。

你可以把大语言模型理解成一个读过海量书籍和代码的超级学霸，它见过无数的编程案例，所以能帮你写代码、解释代码、修复 bug。

### 模型参数



参数是模型在训练过程中学到的 “知识点”，用数字的形式存储在模型中。参数越多，模型能记住的知识就越丰富，通常也越聪明。

比如：

- GPT-4 大约有 1.8 万亿参数
- Claude 3.5 Sonnet 的参数量未公开，但估计在千亿级别
- DeepSeek-V3 有 6710 亿参数

参数量会影响模型的能力和运行成本。一般来说，参数越多的模型越贵，但效果也越好。

### 训练和推理



训练（Training）是让 AI 模型从大量数据中学习知识的过程。这个过程需要海量的计算资源和时间，通常由 AI 公司完成。你不需要自己训练模型。

推理（Inference）是模型学完之后，用学到的知识来回答问题、生成内容的过程。当你用 ChatGPT 对话时，就是在进行推理。

打个比方：训练就像学生上学读书，推理就像学生参加考试答题。我们日常使用 AI 工具，都是在用推理能力。

### 微调（Fine-tuning）



微调是在已有模型的基础上，用特定领域的数据继续训练，让模型在某个领域表现更好。

比如，你可以用大量的医学资料微调一个模型，让它成为医学专家。或者用你公司的代码库微调，让它更了解你的项目风格。

对于普通用户来说，微调成本较高，一般不需要自己做。直接使用现成的模型就够用了。

## Token 和计费



### Token



Token 是 AI 模型处理文本的基本单位。你可以简单理解为 “词块”。

在英文中，一个 Token 大约是一个单词或单词的一部分。在中文中，一个汉字通常是 1-2 个 Token。

为什么 Token 重要？因为 AI 服务通常按 Token 收费。你输入的文字和 AI 输出的文字都会消耗 Token。Token 用得越多，花的钱就越多。

举个例子：

- "Hello World" 大约是 2 个 Token
- “你好世界” 大约是 4-6 个 Token

### 输入 Token 和输出 Token



AI 服务通常分别计算输入和输出的 Token：

- 输入 Token：你发给 AI 的内容（提示词、代码、文件等）
- 输出 Token：AI 返回给你的内容（回答、生成的代码等）

一般来说，输出 Token 比输入 Token 更贵，因为生成内容比理解内容更消耗算力。

省钱小技巧：写清楚、写简洁的提示词，让 AI 一次就能理解你的需求，减少反复对话。

### 上下文窗口



上下文窗口（Context Window）是指 AI 模型一次能 “记住” 的最大内容量，用 Token 来衡量。

不同模型的上下文窗口大小不同：

- GPT-4o：128K Token（约 10 万中文字）
- Claude 3.5 Sonnet：200K Token（约 15 万中文字）
- Gemini 2.0 Pro：2M Token（约 150 万中文字）

上下文窗口越大，AI 能处理的代码量就越多，能记住的对话历史就越长。如果你的项目代码很多，选择上下文窗口大的模型会更合适。

但要注意，上下文窗口越大，每次请求消耗的 Token 也越多，成本也会更高。

## 提示词相关



### 提示词（Prompt）



提示词是你给 AI 的指令或问题。在 Vibe Coding 中，提示词就是你用自然语言描述的需求。

提示词的质量直接决定了 AI 输出的质量。一个好的提示词应该：

- 具体明确，不含糊
- 包含必要的背景信息
- 说明期望的输出格式

比如，“做一个网站” 是一个模糊的提示词，而 “用 React 做一个记账网站，包含添加支出、查看列表、统计总额三个功能，界面用蓝色调” 就是一个好的提示词。

在 AI 对话中，消息通常分为三种角色：

- 系统提示词（System）：设置 AI 的角色和行为规则，对用户不可见
- 用户提示词（User）：你发送给 AI 的消息
- 助手提示词（Assistant）：AI 回复给你的消息

理解这 3 种角色有助于你更好地构造对话。比如在调试时，你可以在提示词中模拟之前的对话历史，让 AI 更好地理解上下文。

### 系统提示词



系统提示词（System Prompt）是在对话开始前设置的指令，用来定义 AI 的角色、行为和限制。

比如，你可以设置系统提示词：“你是一位资深的 React 开发专家，请用简洁清晰的代码风格回答问题。”

系统提示词在整个对话过程中都会生效，是定制 AI 行为的重要方式。

### 提示词工程



提示词工程（Prompt Engineering）是设计和优化提示词的技术，目的是让 AI 更好地理解你的意图，生成更符合预期的结果。

这是 Vibe Coding 的核心技能之一。好的提示词工程师能用更少的对话轮次，让 AI 生成更高质量的代码。

### 零样本提示（Zero-shot）



零样本提示是指直接给 AI 一个任务，不提供任何示例。

比如：“请把这段英文翻译成中文。”

AI 会根据自己的训练知识来完成任务。对于简单任务，零样本提示通常就够用了。

### 少样本提示（Few-shot）



少样本提示是在提示词中提供几个示例，让 AI 学习你想要的格式或风格。

比如：

```
请按以下格式翻译：
英文：Hello → 中文：你好
英文：Thank you → 中文：谢谢
英文：Good morning → 中文：
```



通过提供示例，AI 能更准确地理解你的需求，输出更一致的结果。

### 思维链提示（Chain-of-Thought）



思维链提示是让 AI 一步一步思考问题，而不是直接给出答案。这对于复杂的推理任务特别有效。

你可以在提示词中加上 “请一步一步思考” 或 "Let's think step by step"，AI 就会展示它的推理过程，通常能得到更准确的答案。

在编程中，思维链提示能帮助 AI 更好地理解复杂需求，生成更合理的代码结构。

### Markdown 语言



Markdown 是一种轻量级的文本标记语言，用简单的符号来表示格式。比如用 `#` 表示标题，用 `**文字**` 表示加粗，用 `-` 表示列表。

在 Vibe Coding 中，Markdown 非常重要，因为：

- AI 生成的回答通常是 Markdown 格式
- 项目文档（如 README）用 Markdown 编写
- 规则文件也是 Markdown 格式

学会 Markdown 能让你更好地与 AI 交流，也能写出更规范的项目文档。

## AI 编程模式



### Vibe Coding



Vibe Coding 是由计算机科学家 Andrej Karpathy 在 2025 年 2 月提出的概念。它描述了一种全新的编程方式：通过自然语言和 AI 对话，让 AI 帮你写代码，你只需要描述需求、测试结果、指导方向。

Vibe Coding 的核心理念是：你不需要精通编程语法，只需要能清楚表达你的想法。AI 负责把你的想法变成可运行的代码。

这就像点外卖一样：你告诉外卖平台你想吃什么，餐厅帮你做好送到手上。你不需要会做饭，但要知道自己想吃什么。

### Agentic Coding 智能体编程



Agentic Coding 是指让 AI 像一个自主的 “智能体”（Agent）一样工作，能够自己规划任务、执行操作、验证结果，而不只是被动地回答问题。

在 Cursor 的 Agent 模式中，AI 可以：

- 自动读取和分析多个文件
- 规划实现方案
- 执行代码修改
- 运行测试验证
- 自动修复问题

这比传统的问答式 AI 更强大，因为它能自主完成复杂的多步骤任务。

### 多智能体协作



多智能体协作（Multi-Agent）是指多个 AI 智能体分工合作，共同完成复杂任务。

比如，一个智能体负责设计架构，一个负责写前端代码，一个负责写后端代码，一个负责代码审查。它们像一个团队一样协作。

这两年，多智能体系统正在成为 AI 编程的重要趋势，能够处理更复杂的项目。

### 智能体编排



编排是指协调和管理多个 AI 智能体或 AI 任务的过程，确保它们按正确的顺序和方式工作。

就像乐队指挥一样，编排器决定哪个智能体在什么时候做什么事情，如何传递信息，如何汇总结果。

### Agent Loop 智能体循环



Agent Loop 是 AI 智能体的核心工作机制，描述了智能体如何持续运行来完成任务。

一个典型的 Agent Loop 包括：

1. 感知：获取当前环境信息（读取文件、查看错误等）
2. 思考：分析情况，决定下一步行动
3. 行动：执行具体操作（写代码、运行命令等）
4. 观察：检查行动的结果
5. 循环：根据结果决定是否继续

这个循环会一直进行，直到任务完成或达到终止条件。理解 Agent Loop 能帮你更好地使用 Cursor Agent 等工具。

### ReAct 推理与行动



ReAct（Reasoning and Acting）是一种让 AI 智能体交替进行推理和行动的技术框架。

传统的 AI 要么只思考不行动，要么只行动不思考。ReAct 让 AI 能够：

- 先推理：思考当前情况，制定计划
- 再行动：执行具体操作
- 观察结果：看看行动效果如何
- 继续推理：根据结果调整策略

这种 “思考 - 行动 - 观察” 的循环让 AI 能更可靠地完成复杂任务，是现代 AI 编程工具的核心技术之一。

### 工具调用



工具调用（Tool Use / Function Calling）是让 AI 能够使用外部工具和功能的技术。AI 本身只能生成文字，但通过工具调用，它可以：

- 读写文件
- 执行命令行命令
- 搜索网页
- 调用 API
- 操作数据库

工具调用的工作流程分为 4 步：

1. 识别需求：AI 判断当前任务需要使用工具
2. 选择工具：从可用工具中选择合适的
3. 执行调用：用正确的参数调用工具
4. 整合结果：将工具返回的结果融入回答

### Agent Skills 智能体技能



Agent Skills（智能体技能）是 Anthropic 在 2025 年 10 月推出的开放标准，用于给 AI 智能体扩展特定领域的专业能力。

简单来说，Skill 就是一个包含 `SKILL.md` 文件的文件夹，里面可以放置指令说明、脚本代码、参考资料等。当 AI 遇到相关任务时，会自动加载对应的 Skill 来增强自己的能力。

### BMAD 敏捷 AI 开发方法



[BMAD-METHOD](https://github.com/bmad-code-org/BMAD-METHOD)（Breakthrough Method of Agile AI-Driven Development，突破性敏捷 AI 驱动开发方法）是一套系统化的 AI 智能体开发框架，旨在将原本混乱的 AI 编程过程变得结构化、可复用。

- Analyst Agent（分析师）：创建项目简报，包含市场分析和用户画像

每个智能体都有标准化的组成部分：人设（角色、身份、沟通风格、原则）、能力列表、交互菜单，以及可选的关键行动。

## 上下文管理



### 上下文（Context）



上下文是 AI 在回答问题时能够参考的所有信息，包括：

- 当前对话的历史
- 你打开的代码文件
- 项目的结构和配置
- 你提供的参考资料

上下文越丰富、越相关，AI 生成的代码就越符合你的需求。这就像给一个新同事交接工作 —— 你给的背景信息越多，他上手就越快。

### 上下文工程



上下文工程（Context Engineering）是有策略地管理和优化提供给 AI 的上下文信息的技术。

核心目标是 **让 AI 拥有恰到好处的信息**。既不能太少（导致 AI 不了解情况），也不能太多（导致信息过载、成本上升）。

好的上下文工程包括：

- 选择最相关的文件
- 提供必要的背景说明
- 使用规则文件定义项目规范
- 适时清理无关的对话历史

### 规则文件



规则文件（Rules File）是放在项目中的配置文件，用来告诉 AI 你的项目规范、技术栈、代码风格等信息。有了规则文件，AI 每次生成代码时都可以参考这些规则，生成的代码更符合你的项目风格，省去了反复强调的麻烦。

不同 AI 编程工具使用不同的规则文件格式：

- Cursor：早期使用 `.cursorrules` 单文件格式，现在推荐使用 `.cursor/rules/*.mdc` 多文件格式
- Claude Code：使用 `CLAUDE.md` 文件
- GitHub Copilot：使用 `.github/copilot-instructions.md` 文件

以 Cursor 为例，现代的 `.mdc` 规则文件支持 YAML 元数据，可以指定规则的适用范围：

```
---
description: React 组件开发规范
globs: src/components/**/*.tsx
alwaysApply: false
---
# React 规范
- 使用函数式组件
- 优先使用 hooks
```



规则文件的激活方式有多种，比如：

- 始终生效：设置 `alwaysApply: true`
- 模式匹配：当引用匹配 `globs` 的文件时自动激活
- 手动调用：在对话中用 `@规则名` 引用
- AI 自主决定：AI 根据任务相关性自动加载

💡 注意，随着工具版本的更新，这些文件的名称和标准可能会发生改变，一切以工具官方文档为主。

### AGENTS.md



[AGENTS.md](https://agents.md/) 是一种开放的文件格式，专门用于给 AI 编程智能体提供项目指令。

传统的 README.md 是写给人看的，主要介绍项目是什么、怎么用。而 AGENTS.md 是写给 AI 看的，包含 AI 工作时需要的技术细节：

- 项目的构建和启动命令
- 测试运行方式
- 代码风格和规范
- 项目结构说明

一个典型的 AGENTS.md 文件大概长这样：

```
# 项目设置

- 启动开发：npm run dev
- 运行测试：npm test

- 使用 TypeScript 严格模式
```

AGENTS.md 的优势在于它是一个开放标准，被数万个开源项目采用。当你使用支持该标准的 AI 编程工具（如 Claude Code、OpenAI Codex、Cursor、GitHub Copilot 等）时，AI 会自动识别项目根目录下的 AGENTS.md 文件，并将其中的指令发送给 AI，无需你手动引用。

### SDD 规范驱动开发



SDD（Spec-Driven Development，规范驱动开发）是 AI 时代的一种新型开发方法论，强调在编码之前先创建精确、机器可读的规范文档。

传统开发流程是：想到什么写什么，边写边改，最后再补文档。这样容易导致需求不清晰、代码和文档对不上。

而 SDD 的思路正好相反：**先把需求写成规范文档，并且把规范文档当作代码的唯一真相来源**。

你可以把规范文档理解为 “项目宪法”，它包含了详细的需求描述、系统设计和接口定义。AI 必须严格遵守这些条文来生成代码，确保产出完全符合预期。

为什么 SDD 越来越受重视？

因为 AI 生成代码的质量直接取决于上下文的清晰度，而不仅仅是依靠提示词技巧。一个清晰的规范文档能比任何 Prompt 黑魔法更有效地减少错误。

SDD 的典型工作流程：

1. Constitution（制定准则）：定义项目的基本原则、代码规范、性能标准
2. Specify（编写规范）：描述要做什么功能、为什么做、用户需求是什么
3. Clarify（澄清疑问）：让 AI 提出结构化问题，明确边界情况和错误处理
4. Plan（制定方案）：确定技术栈、系统架构、数据模型、API 接口
5. Tasks（拆解任务）：把计划拆解成可执行的任务列表，标注依赖关系和优先级
6. Implement（执行实现）：AI 按照任务列表生成代码，人类验证

### RAG 检索增强生成



RAG（Retrieval-Augmented Generation）是一种让 AI 能够查阅外部知识库的技术。

### 向量数据库



向量数据库是专门用来存储和查询 “向量”（一种数字表示形式）的数据库。在 AI 领域，它常用来存储文本的语义表示。

### MCP 模型上下文协议



MCP（Model Context Protocol）是 Anthropic 在 2024 年底推出的开放标准，用于让 AI 模型安全地连接外部数据源和工具。

你可以把 MCP 理解成 AI 世界的 “USB 接口”。有了 MCP，AI 就能方便地读取你的文件、访问数据库、调用各种工具，而不需要每个工具都单独开发接口。

## AI 输出相关



### AI 幻觉



AI 幻觉（Hallucination）是指 AI 编造了不存在的内容，比如虚构的 API、错误的函数用法、不存在的库。

这是大语言模型的固有问题，因为它们是基于概率生成内容的，有时会 “脑补” 不存在的东西。

遇到幻觉时的应对方法：

- 要求 AI 提供文档链接验证
- 自己查官方文档确认
- 换个模型试试
- 开新对话重新描述问题

### 温度



温度（Temperature）是控制 AI 输出随机性的参数，取值通常在 0-2 之间。

- 温度低（如 0.1）：输出更确定、更保守，适合写代码
- 温度高（如 1.0）：输出更随机、更有创意，适合头脑风暴

在编程场景中，通常使用较低的温度，让 AI 生成更稳定、更可预测的代码。

### 流式输出



流式输出（Streaming）是指 AI 一边生成内容，一边实时显示给你，而不是等全部生成完再显示。

这就像看直播而不是看录播，你能实时看到 AI 的思考过程，如果发现方向不对，可以及时中断。

大多数 AI 编程工具都支持流式输出，让交互体验更流畅。

## 项目管理概念



### MVP（最小可行产品）



MVP（Minimum Viable Product）是用最少的功能满足核心需求的产品版本。

做 MVP 的好处是：

- 快速验证想法是否可行
- 避免在不必要的功能上浪费时间
- 更快获得用户反馈

比如做记账应用，MVP 版本可能只有记录支出、查看列表两个功能，其他高级功能以后再加。

### 迭代开发



迭代开发是把大项目分成多个小周期，每个周期完成一部分功能的开发方法。

每个迭代周期包括：计划 => 开发 => 测试 => 发布 => 反馈 => 改进。

这种方法特别适合 Vibe Coding，因为你可以让 AI 先实现核心功能，测试没问题后再逐步添加新功能。

### 重构



重构（Refactoring）是在不改变功能的前提下，改进代码结构和质量的过程。

重构的目的是让代码更清晰、更易维护、更高效。常见的重构包括：

- 提取重复代码为函数
- 改进变量和函数命名
- 简化复杂的逻辑
- 拆分过长的文件

在 Vibe Coding 中，你可以让 AI 帮你重构代码，但要小步进行，每次重构后都要测试。

### 技术债



技术债（Technical Debt）是为了快速完成功能而采用的临时方案，这些方案在未来需要花时间修复和改进。

就像信用卡欠款一样，虽然现在透支方便，但以后要还，还要加利息。

在 Vibe Coding 中，AI 生成的代码可能不是最优方案，积累太多技术债会让项目越来越难维护。定期重构是偿还技术债的方式，防止出现屎山代码。

## 前后端概念

前端（Frontend）是用户能直接看到和交互的部分，包括网页界面、按钮、表单、动画等。

在 Vibe Coding 中，前端是 AI 最擅长生成的部分，因为效果可以直接看到，方便验证和调整。

## 三、提示工程（Prompting）

### 3.1 基础技巧

- **角色与任务描述**：系统提示（system prompt）与用户提示的划分。
- **Few-shot / Zero-shot**：示例数量与质量对效果的影响。
- **链式思考（CoT）**：Let’s think step by step、分步推理与中间结果利用。
- **格式约束**：JSON、Markdown、XML 等结构化输出与解析。

### 3.2 进阶与鲁棒性

- **思维链变体**：Tree of Thought、Self-Consistency、Least-to-Most。
- **提示注入与防御**：边界划分、敏感指令过滤、输出校验。
- **多轮对话**：历史管理、上下文截断与摘要策略。
- **多语言与领域**：领域术语、少样本示例、角色一致性。

## 四、RAG（检索增强生成）

### 4.1 流程与组件

- **整体流程**：查询理解 → 检索 → 重排序/筛选 → 拼进提示 → 生成 → 可选的后处理。
- **文档处理**：切块策略（按段落/按长度/按语义）、重叠、元数据。
- **向量化**：Embedding 模型选择、维度、归一化与距离度量（余弦、欧氏）。
- **检索**：稠密检索（向量库）、稀疏检索（BM25）、混合检索（Hybrid）与融合方式。

### 4.2 向量库与检索优化

- **向量数据库**：Milvus、Pinecone、Weaviate、pgvector、Elasticsearch 等；选型考量（规模、延迟、过滤）。
- **重排序（Reranker）**：两阶段检索、Cross-Encoder、节省 token 与提升准确率。
- **查询改写与扩展**：同义词、多查询、假设性文档嵌入（HyDE）。
- **评估**：检索 recall@k、MRR、NDCG；端到端答案相关性、 faithfulness、引用准确性。

### 4.3 常见问题与优化

- **幻觉与溯源**：引用来源、可验证性、无答案时的降级策略。
- **长文档与多文档**：长上下文模型 vs 分块 RAG、文档级去重与排序。
- **实时与更新**：增量索引、失效策略、多版本文档。

### 5.1 Agent 基础概念

#### 1. Agent 的通用抽象架构与“感知、规划、行动、记忆”闭环

**通用抽象架构**：Agent 通常被抽象为在环境中**自主决策、执行动作**的实体，其核心是“感知 → 规划 → 行动 → 观察”的闭环，再叠加**记忆**形成跨轮次、跨任务的一致性。

- **感知（Perception）**：从环境或用户获取输入，包括当前轮的用户消息、工具返回的 Observation、系统状态等。
- **规划（Planning）**：基于当前状态与目标，决定“下一步做什么”——是直接回复、调用工具、还是拆解子任务。规划可能是一次性（Plan-and-Execute）或逐步迭代（ReAct）。
- **行动（Action）**：执行具体动作，如调用工具、生成回复、写入记忆等。
- **记忆（Memory）**：存储历史对话、关键事实、用户偏好、任务中间结果等，供后续感知与规划使用。

**闭环协作**：  
感知提供“当前状态”，规划基于“状态 + 记忆”做出决策，行动改变环境或产生新信息，新信息作为下一轮的感知输入，同时选择性写入记忆。记忆使 Agent 在长对话或多步任务中保持上下文一致、避免重复与矛盾。

#### 2. LLM 的“无状态”与“上下文窗口限制”及主流记忆方案

**问题**：  
- **无状态**：每次调用 LLM 时，模型不天然记住上次对话，只能依赖本次请求内的上下文。  
- **上下文窗口限制**：输入+输出总 Token 有上限，历史过长无法全部放入，且长上下文会带来注意力分散、成本与延迟上升。

**主流解决方案对比**：

| 方案 | 思路 | 优点 | 缺点 |
|------|------|------|------|
| **Full History** | 把所有历史消息都塞进上下文 | 信息完整、实现简单 | 易超窗口、成本高、长文本时注意力衰退 |
| **Sliding Window** | 只保留最近 N 轮/ N 条消息 | 控制 Token、实现简单 | 早期关键信息丢失 |
| **Summary Memory** | 对较早历史做摘要，保留摘要+近期原文 | 平衡长度与信息保留 | 摘要可能丢失细节、摘要质量依赖模型 |
| **Vector Store Memory** | 历史关键信息向量化存储，按需检索注入 | 可扩展、支持长历史、按相关性注入 | 依赖检索质量、可能有遗漏或噪声 |

实际系统常**组合使用**：近期用 Sliding Window，更早的用 Summary 或 Vector 检索后按需注入；关键事实也可单独写入“长期记忆”再检索。

### 5.2 ReAct 与推理范式

#### 3. 什么是 ReAct 范式？为何能显著提升 LLM 解决复杂任务的能力？

**ReAct**（Reasoning + Acting）是一种让 LLM **交替进行推理与行动**的范式：模型先给出一步推理（Thought），再决定并执行一个行动（Action），获得观察（Observation）后继续推理，如此循环直到得出最终答案。

- **推理轨迹（Reasoning Trace）**：  
  显式写出“当前状态—为何要这么做—选哪个工具—预期得到什么”，使推理过程可追溯、可纠错；同时把“中间结论”写进上下文，减轻遗忘与幻觉。

- **行动空间（Action Space）**：  
  把“可选动作”限定为有限的工具调用（如 search、calculator、run_code），模型只需生成工具名+参数，而不是自由文本，既降低幻觉又便于程序执行与观测。

**为何能提升复杂任务能力**：  
复杂任务需要多步推理与多步操作；纯 CoT 只推理不行动，无法获取真实世界反馈；纯工具调用缺少连贯推理易乱试。ReAct 把“思考下一步”和“执行并观察”绑在一起，用观察结果纠正推理，形成闭环。

#### 4. ReAct 循环中的“死循环”与“幻觉行动”及算法层打破策略

**死循环**：同一或相似 Action 反复执行、始终达不到终止条件。  
**幻觉行动**：调用不存在的工具、参数格式错误、或基于错误推理做出无效动作。

**算法层打破策略**（不限于 ReAct，但 ReAct 中最常用）：

- **步数上限与熔断**：设定最大步数，超过则强制结束并返回“未完成”或走降级路径。
- **重复检测**：记录近期 N 步的 (Action, 关键参数)，若检测到重复则禁止再执行相同动作，或强制进入“反思/换策略”分支。
- **Observation 有效性检查**：若连续多次 Observation 为错误/超时/空，则触发重试或换工具/换参数，而不是继续同一路径。
- **反思（Reflection）**：在达到步数阈值或检测到重复时，插入一步“反思”：让模型总结已做过的尝试、失败原因，并输出“下一步应避免什么、应尝试什么”，再继续推理。
- **约束解码与 Schema 校验**：工具名与参数严格按 JSON Schema 生成并校验，非法则重试或返回错误信息给模型，减少“幻觉行动”的执行机会。

#### 16. ReAct 中“Reasoning”与“Acting”如何交替？何时失效、如何改进？

**交替方式**：  
典型流程为：  
1）输入：当前任务 + 历史 (Thought, Action, Observation) 序列；  
2）模型输出一段 **Thought**（对当前状况的分析、下一步计划）；  
3）模型输出 **Action**（工具名 + 参数）；  
4）执行工具得到 **Observation**；  
5）将 (Thought, Action, Observation) 追加到上下文，回到步骤 2；  
6）若模型输出“最终答案”而非 Action，则结束。

**失效场景**：  
- 任务需“先全局规划再执行”，ReAct 的逐步试错成本高、易偏离目标。  
- 工具依赖复杂（如 A 的输出是 B 的输入），模型容易搞错顺序或重复调用。  
- 观察噪声大（如搜索结果无关），模型容易被误导进入歧途。

**改进方向**：  
- 引入 **Plan-and-Execute**：先让模型输出多步计划，再按步执行并在执行中允许局部重规划。  
- **子目标与检查点**：把长任务拆成带检查点的子目标，每段用 ReAct，段间用规划或人工确认。  
- **更好的工具描述与示例**：减少误选工具、误填参数。  
- **反思与重试**：在关键步骤或失败后插入反思，让模型总结后再继续。

### 5.3 规划与执行、目标拆解

#### 6. Plan-and-Execute 与 ReAct 的异同及适用场景

| 维度 | ReAct | Plan-and-Execute |
|------|--------|-------------------|
| 顺序 | 边推理边行动，每步都可依赖上步观察 | 先规划出多步计划，再逐步执行 |
| 规划 | 隐式、逐步、可随时调整 | 显式、一次性或阶段性规划 |
| 适应性 | 高，每步可根据观察调整 | 执行阶段相对固定，依赖“重规划”弥补 |
| 可控性 | 路径不可预测 | 计划可审查、可人工干预 |

**Plan-and-Execute 更优的场景**：  
- 任务步骤清晰、可预先列出（如“先查 A 再查 B 再汇总”）。  
- 需要人类在关键步骤审批或修改计划。  
- 希望执行轨迹可解释、可审计。  
- 工具链长且依赖关系明确，一次性规划能减少无效尝试。

#### 10. CoT 与 ToT 在复杂推理中的路径差异

- **CoT（Chain of Thought）**：单一路径的逐步推理，一步接一步，不回溯。适合有明确推理链的问题；错误一旦出现会延续。
- **ToT（Tree of Thoughts）**：把“下一步”扩展成多个候选（多叉），对候选做评估或剪枝，再沿较优分支继续扩展，形成树状搜索。适合多解、需探索或易错的推理；通过回溯和评估可避免单路径错误。

**路径差异**：CoT 是链；ToT 是树，通过评估与剪枝在树中选路径，计算与 Token 成本更高，但复杂推理更稳。

#### 65. Plan-and-Execute 中“重规划”机制

执行中途发现计划不可行时（如某步失败、前提不成立、用户改需求），可触发**重规划**：  
- 输入：当前状态、已执行步骤、失败/变更原因。  
- 由模型生成新的剩余计划（或全量计划），可保留“仍有效的步骤”、只重排或替换失效部分。  
- 再按新计划继续执行，并可设重规划次数上限以防无限重试。

### 5.4 约束解码与工具调用

#### 5. 约束解码在 Agent 中的作用；如何通过强制 Schema（如 JSON）提高工具调用成功率

**作用**：  
约束解码限制模型**输出格式**必须符合给定语法（如 JSON、特定 tool_use 块），从而保证：  
- 输出可直接解析为工具名+参数，减少格式错误导致的调用失败。  
- 避免模型“编造”工具名或参数结构（减少幻觉调用）。

**提高成功率的方式**：  
- 在 System/User 中明确工具列表及每个工具的 **JSON Schema**（参数名、类型、必填/可选）。  
- 使用 **JSON Mode** 或 **Grammar-based decoding**（如 GBNF）：在解码时只允许生成合法 JSON，这样字段缺失、类型错误、多余逗号等问题可从根上减少。  
- 若无法约束解码，则**后处理校验**：解析失败时重试或返回错误信息给模型让其修正。

#### 17 / 58. Function Call 的底层实现原理；模型如何被训练以支持 tools 参数

**使用形态**：  
调用 Chat Completion API 时传入 `tools`（工具列表，含 name、description、parameters 的 JSON Schema），模型在回复中可返回 `tool_calls`（工具名 + 参数），再由应用执行工具并把结果以 `tool` 角色送回，模型据此继续生成。

**底层实现思路**：  
- **训练**：在指令微调数据中加入“多轮对话 + 工具调用”样本：用户请求 → 模型输出 tool_calls → 工具结果 → 模型最终回复。模型学习“何时该调用哪个工具、参数如何填”。部分方案还会在 token 层面对 tool_use 做特殊标记或约束。  
- **推理**：模型在生成时预测“下一段是普通文本还是 tool_use”；若为 tool_use，则输出被解析为 JSON（或通过约束解码保证合法），再由运行时根据 schema 校验并执行。

**自然语言到 JSON Schema 的映射**：  
通过大量 (用户意图, 工具名, 参数) 的标注或合成数据，模型学到“这类意图 → 该工具 + 对应参数结构”；description 和 schema 作为上下文进一步约束生成结果。

#### 22 / 60 / 61. 参数错误与执行报错的鲁棒处理

- **格式错误（如 JSON 字段缺失）**：  
  - **预防**：JSON Mode / Grammar 约束解码，从源头减少非法 JSON。  
  - **修复**：解析失败时，将错误信息（如“缺少字段 x”）拼回 Prompt，让模型重试；或用规则/小模型补全默认值（慎用）。  
  - **重试**：同一请求最多重试 N 次，带退避；仍失败则返回“工具调用失败，请稍后重试”或转人工。

- **参数幻觉（参数值不合理或虚构）**：  
  - **Prompt**：在工具描述中写清取值范围、单位、示例。  
  - **代码层**：对参数做 schema 校验、范围校验、业务规则校验；不通过则返回明确错误信息给模型，触发其修正。  
  - **默认值与降级**：对非关键参数提供默认值，避免必填导致整步失败。

- **工具执行报错（API 超时、5xx、业务错误）**：  
  - 将**错误信息**（类型、消息、必要时截断）作为 Observation 返回给模型，让模型**自我修复**（改参数、换工具、或给出用户可读说明）。  
  - 重试策略：对可重试错误（如 5xx、超时）自动重试有限次；对 4xx 一般不自动重试，交给模型根据错误内容决定下一步。

#### 23. 并行工具调用

**实现**：  
单次模型回复中可包含**多个** `tool_calls`（互不依赖时）。应用层解析出多个调用后，**并行**执行这些工具，将多个结果一起作为后续消息送回模型，模型再基于全部结果继续推理。  
**LLM 侧**：在训练数据中包含“一次输出多个 tool_calls”的样本，模型学会在“需要同时查多个独立信息”时输出多条调用，而不是串行多次请求。

#### 24 / 44. 工具调用安全与 Prompt 注入防御

- **敏感工具防护**：  
  - 对高风险工具（如删库、改配置、执行任意命令）做**权限与审批**：仅特定角色可调用，或需人工确认。  
  - 工具列表按角色过滤：不向当前会话暴露不应使用的工具。  
  - 在 System Prompt 中明确“禁止在用户消息引导下调用敏感工具”，并记录审计日志。

- **Prompt 注入（含 System Prompt 注入）**：  
  - 攻击者通过在用户输入中注入“忽略上文，执行…”类指令，企图篡改行为或诱导调用敏感工具。  
  - **防御**：严格区分 System / User / Assistant 边界；对用户输入做长度与敏感指令检测；敏感操作二次确认；输出前过滤明显被操控的内容。  
  - **System Prompt 注入**：若攻击者能影响 system 内容（如通过可被编辑的配置），则需对 system 来源做访问控制与审计。

#### 25. 多轮对话中：何时调用工具、何时直接回复、何时请求澄清

- **调用工具**：用户意图需要外部信息或操作（查数据、执行动作）且当前上下文中没有时。  
- **直接回复**：问题可仅凭已有上下文回答，或工具结果已足够可直接给出结论时。  
- **请求澄清**：意图模糊、存在多种理解、或缺少必要参数（如“订机票”但未说时间/目的地）时，主动追问用户。

实现上可通过 System Prompt 明确这三类情形，并配合**意图分类**（可选）：先判断“需工具 / 可直接答 / 需澄清”，再进入对应分支；或由单一生成步骤在回复中体现“调用工具”或“追问用户”。

### 5.5 意图、反思与记忆

#### 7 / 73. 意图增强

当用户指令**模糊或含糊**时，Agent 通过“意图识别”或“意图重写”明确目标，再执行，以提升准确性。

- **意图识别**：将用户输入分类到预定义意图（如查询、订票、投诉），再按意图路由到不同工具或流程。  
- **意图重写**：用 LLM 将用户短句扩展为结构化、无歧义的任务描述（如补全缺失的时间、对象、范围），再交给规划/工具模块。  
- **实现**：在 System Prompt 中要求“若用户意图不明确，先列出可能理解并请求用户确认，或根据上下文做最合理重写”；也可单独调用一个小模型做意图分类/重写，再主模型执行。

#### 8 / 15. 反思机制与 Self-Reflection

**反思的作用**：  
在关键步骤或失败后，让模型对“已做过的行动与结果”进行总结与评价，从而调整下一步策略，减少重复错误、避免死循环。

**是否必须依赖外部反馈**：  
不必须。**自我反思**仅依赖模型自身输出与工具返回的 Observation，不依赖人工标注或外部奖励信号。例如：让模型在每 N 步或检测到重复时，生成一段“反思”：哪些尝试失败了、原因是什么、下一步应避免什么、应尝试什么，再把这段反思加入上下文继续推理。  
**高效 Reflection Prompt 设计要点**：  
- 明确输入：当前目标、已执行步骤列表、各步结果与错误信息。  
- 明确输出格式：例如“失败原因总结 + 下一步建议（1–3 条）”。  
- 限制长度，避免反思占用过多 token。

#### 14 / 95. 记忆在智能体中的角色；短期/长期记忆及分层

**角色**：  
记忆为 Agent 提供跨轮次、跨步骤的**状态与知识**，使行为一致、可延续，避免重复询问和重复尝试。

- **短期记忆**：当前会话的对话历史、本任务内的 Thought/Action/Observation 序列。实现：Sliding Window、或 Full History（在窗口内）。  
- **长期记忆**：跨会话或跨任务需要保留的信息（用户偏好、关键事实、任务结果摘要）。实现：向量库、知识库、或结构化存储，按需检索注入。  
- **分层/闪存**：如“工作记忆（当前任务最近几步）+ 短期（本会话）+ 长期（向量/DB）”。数据流转：重要结论或用户事实写入长期；当前任务进行中的中间状态放在工作/短期；检索时按相关性从长期取回，与短期拼接后送入模型。

### 5.6 上下文工程与检查点

#### 11. 上下文工程 vs 传统 Prompt Engineering；System Message 核心要素

**区别**：  
- **Prompt Engineering** 侧重单轮或少量轮次的提示设计（如何写问法、Few-shot、格式）。  
- **上下文工程** 侧重**整段对话中放什么、不放什么、如何组织**：历史如何截断、摘要、检索注入；系统提示、工具描述、当前输入如何排版；如何在有限窗口内最大化有用信息。

**System Message 常见核心要素**：  
- 角色与行为约束（你是谁、能做什么、不能做什么）。  
- 工具列表及使用说明（名称、描述、参数 schema）。  
- 输出格式要求（如 JSON、Markdown、必须包含的字段）。  
- 安全与边界（不回答敏感问题、不执行未授权操作）。  
- 可选：Few-shot 示例、当前会话的上下文策略说明（如“请基于上述对话与检索结果回答”）。

#### 56. 元提示（Meta-Prompt）：自动生成子 Prompt

针对复杂任务，可设计**元提示**让 Agent 先为该任务生成“定制化子 Prompt”：输入为任务描述与约束，输出为一条或多条专门用于执行该任务的 prompt（如检索策略、输出格式、步骤约定）。执行阶段再使用这些子 prompt 调用 LLM 或子 Agent，使单次通用 prompt 变为“任务专属 prompt”，提升执行准确性。

#### 74. 有限 Context Window 下的记忆管理策略

在窗口有限时，常用三种策略（可组合）：  
- **压缩/摘要**：对较早历史做摘要，只保留摘要 + 近期原文。  
- **筛选/检索**：不按时间保留，而是按**相关性**从历史中检索与当前最相关的片段注入（类似 Vector Store Memory）。  
- **分层**：工作记忆（当前任务最近几步）+ 短期（本会话摘要）+ 长期（向量库/DB），按需从长期取回与短期拼接，控制总 token。

#### 12 / 53 / 76. 检查点与长对话策略；上下文压缩

**检查点**：  
在长链条任务中，在关键步骤完成后**持久化状态**（如当前计划、已完成的子目标、中间结果）。若后续失败或超时，可从最近检查点恢复，无需从头重跑。实现：将“计划 + 已完成步骤 + 当前输出”写入 DB 或文件，并支持“从检查点 ID 恢复”的入口。

**长对话策略**：  
- 保留最近 N 轮完整对话 + 更早的摘要或检索结果。  
- 对超长单条消息做摘要或分段检索后再注入。  
- 明确“关键信息”的抽取与保留（如用户明确说过的偏好、约束），单独存储并在后续轮次优先注入。

**上下文压缩**：  
在保留率与 Token 成本间权衡：摘要压缩、按重要性过滤、向量检索只取 Top-K、或分层（摘要 + 近期原文），避免无差别截断导致丢失关键信息。

#### 54. 长上下文中注意力衰退及工程应对

**现象**：  
序列变长时，模型对**中间或前部**内容的注意力权重下降，更多依赖**尾部**信息，导致早期关键信息被“遗忘”。

**工程方案**：  
- **Sliding Window**：只保留最近一段，控制有效长度。  
- **向量检索混合**：长文档不整篇输入，先检索相关片段再注入，相当于“重要部分”被放到尾部附近。  
- **分层摘要**：长文分成块，每块摘要后再拼成较短上下文；或多级摘要（块摘要 → 段摘要），只把最相关层注入。

### 5.7 工具学习、描述与路由

#### 9. Toolformer 核心思想

**核心思想**：  
让模型在**少量或零**人工标注“何时调哪个 API”的情况下，学会**自发**调用外部 API。做法大致为：在预训练或指令数据中，将“API 文档 + 示例调用与结果”作为上下文，模型学习预测“此处应插入 API 调用”；通过自监督或少量标注，模型学会在合适时机生成 tool call 并利用返回结果继续生成。这样工具使用能力可随文档与示例扩展，而不必为每个 API 做大量标注。

#### 20. 工具描述的详细程度与 Token 消耗；对 Zero-shot 工具调用的影响

- **详细**：参数说明全、示例多、边界情况写清 → 模型更易选对工具、填对参数，但占用 token 多，工具多时无法全量注入。  
- **简略**：省 token，但模型易误解或幻觉参数。  
- **平衡**：核心工具给足描述与示例；大量工具时用**分类/分层**，先选类再选具体工具，或只注入与当前意图相关的工具子集（见下条）。  
- **质量**：描述清晰、示例正确能显著提升 Zero-shot 调用成功率；含糊或错误示例会误导模型。

#### 21. 大量工具下的路由：向量召回 + 模型推理

- **问题**：成百上千个工具无法全部放进上下文，需要先**召回**一小部分再让模型选择。  
- **方案**：  
  - **向量召回**：对工具名称+描述做 embedding，用当前用户意图或上一轮 query 的向量检索 Top-K 个工具，只把这 K 个的 schema 注入上下文。  
  - **模型推理**：在召回的小集合上，由 LLM 根据任务选择具体工具并填参。  
- 可再加一层**分类**：先粗分类（如“搜索 / 计算 / 写文件”），再在类内做向量检索或模型选择，进一步减少候选集。

#### 26. LangChain 的 Tool 与 StructuredTool

- **Tool**：基础接口，通常由一个函数 + name/description 组成，Agent 通过描述选择并调用。  
- **StructuredTool**：参数由 **Pydantic 或 JSON Schema** 定义，便于校验与生成时约束，减少参数幻觉。  
- **快速封装**：用 `@tool` 装饰器或 `StructuredTool.from_function` 将普通 Python 函数转为 Tool；为参数写好 schema 即可被 Agent 与 LLM 使用。

#### 27. 工具学习 Agent（能学习新工具用法）

思路：Agent 不仅使用预定义工具，还能通过**阅读文档**或**试错**学习新工具。  
- **阅读文档**：把工具文档（API 说明、示例）放入上下文或检索库，在“需要新能力”时检索相关文档，让模型根据文档生成调用方式并执行；失败时把错误信息与文档片段一起再喂给模型修正。  
- **试错**：在沙箱中执行，根据返回错误调整参数或调用方式，并可选地将“成功调用模式”写入长期记忆供后续使用。

#### 55. 多模态 Agent 中图像、文本、音频的上下文对齐

**挑战**：不同模态的 token 化与位置不同，模型需要对齐“同一时刻”的多模态输入。  
**常见做法**：统一表示（多模态编码器或 VLMs 将图/音转为与文本同空间）；在训练或 prompt 中建立模态间对应；长多模态做采样或摘要，只注入关键帧/片段以控制 token。

#### 28. 虚构工具调用的规避

**现象**：模型输出不存在的工具名、或错误参数结构。  
**规避**：  
- **约束解码**：只允许生成预定义工具名及符合 schema 的参数（如 JSON Schema + Grammar decoding）。  
- **后处理**：解析出 tool_calls 后校验 name 是否在允许列表、参数是否通过 schema；不通过则丢弃该调用并将错误信息返回模型重试。  
- **Prompt**：明确列出“仅可使用以下工具”，并强调“不要编造工具名”。

### 5.8 协议与框架：MCP、A2A、LangChain、LangGraph、Dify

#### 18 / 86. MCP（Model Context Protocol）；Host、Client、Server

**MCP** 是让 AI 模型**标准化接入**外部数据源与工具的协议，解决传统 Function Call 集成中“每家 API 格式不同、重复开发适配”的痛点。

- **Host**：运行 AI 应用或 Agent 的环境（如 IDE、Web 应用），负责发起请求、展示结果。  
- **Client**：由 Host 内嵌，按 MCP 协议向 Server 请求“资源/工具列表、调用工具”等。  
- **Server**：对外暴露资源（如文件、数据库、API）和工具，实现 MCP 服务端接口，可独立进程或跨机部署。

**架构关系**：Host 内运行 Client；Client 通过标准协议（如 stdio/HTTP/SSE）与一个或多个 Server 通信；Server 提供统一的“工具与资源”抽象，便于不同 Host 复用同一套数据与能力。

#### 19. A2A（Agent-to-Agent）协议

**设计初衷**：  
不同框架、不同实现的 Agent 之间需要**互操作**：统一消息格式、能力描述、任务委托与结果回传。A2A 定义 Agent 间通信的协议（如请求/响应结构、能力发现、会话标识），使多 Agent 协作不依赖单一厂商或框架。

**解决互操作性**：  
约定“谁在什么时机发什么消息、如何表示任务与结果”，不同团队开发的 Agent 只要实现同一协议即可组网协作。

#### 29 / 63. LangGraph 与 LangChain 的区别；为何 LangGraph 更适合有状态 Agent

- **LangChain**：以 **Chain** 为主，将组件（LLM、Tool、Retriever 等）串成固定或条件分支的流水线，状态多在链间通过变量传递，复杂循环与多轮决策需要自己维护状态，较难表达“循环、分支、人工介入”。  
- **LangGraph**：以**图**为模型，节点为计算步骤（如调用 LLM、执行工具），边为状态转移条件（如“若需工具则转到工具节点”）。图可包含环，天然支持“思考 → 行动 → 观察 → 再思考”的 ReAct 循环，且**状态**作为图的一等公民在节点间流转，更适合有状态、多步、可中断的 Agent。

**状态图概念**：  
图的每个节点读/写**共享状态**（如 messages、current_step）；边可根据状态字段决定下一节点（如 `has_tool_calls -> tool_node`）。比传统 DAG 更灵活：可循环、可条件分支、可挂“人工确认”节点。

#### 30. LangGraph 中人机协同节点

在关键步骤插入**暂停节点**：该节点不直接执行 LLM/工具，而是将当前状态（如计划、待确认的操作）输出给前端，并**等待**人工输入（确认/拒绝/修改）。实现上：该节点将状态写入“等待人工”的槽位，图执行挂起；前端轮询或 WebSocket 收到“待确认”状态后展示给用户，用户操作后向图注入新事件（如 `user_approved`），图从该节点继续向下执行。

#### 36. LangChain AgentExecutor 中 Observation 与 Thought 的循环

在 `AgentExecutor._call` 中，典型流程为：  
1）用当前输入 + 历史构造 prompt，调用 LLM；  
2）解析 LLM 输出：若包含 `tool_calls` 或解析出的 Action，则执行工具，得到 **Observation**；  
3）将 (Thought/Action, Observation) 追加到消息历史；  
4）若解析出“最终答案”则返回；否则回到 1）。  
即：每次迭代 = 一次 LLM 调用（产生 Thought + 可选 Action）+ 可选工具执行（产生 Observation），直到模型输出结束标记或达到最大步数。

#### 34 / 64. Dify 任务编排；Workflow vs Chatflow；自定义 Plugin

- **任务编排**：Dify 将“节点+边”的可视化编排解析为**内部 DSL**（或图结构），再转化为对 LLM、工具、条件分支的调用序列；执行引擎按图遍历节点，处理变量与分支。  
- **Workflow vs Chatflow**：Workflow 多为固定流程的自动化任务（如定时跑某流程）；Chatflow 以对话为主线，根据用户输入走不同分支，更强调多轮与上下文。底层可能共用同一套节点类型与执行引擎，但触发方式与状态管理不同。  
- **自定义 Plugin**：通过 Dify 的插件接口（如自定义节点、自定义工具）扩展能力；实现约定接口并注册后，即可在编排中使用。

#### 31. AutoGPT 的长期记忆与文件系统

**长期记忆**：AutoGPT 将重要信息写入**本地文件**（如 JSON、Markdown），作为“外挂记忆”；后续轮次通过读取这些文件或摘要后注入上下文，实现跨会话记忆。  
**文件系统**：通过工具暴露“读文件/写文件/列目录”，Agent 可创建任务列表、中间结果、总结等文件，用文件系统持久化状态并在需要时加载，相当于用文件系统充当长期记忆的存储层。

#### 32. MetaGPT 的消息池与多智能体信息共享

**消息池**：多 Agent 不直接两两通信，而是向**共享消息池**发布消息（带角色、类型、内容）；各 Agent 从池中按需订阅或拉取与自己相关的消息。这样解耦发送方与接收方，便于扩展与审计。  
**信息共享**：通过统一的消息格式与路由规则（如“产品经理发布 PRD，架构师只消费 PRD 类消息”），实现多智能体间的信息共享与协作顺序控制。

#### 33. AgentScope 的分布式 Agent

**分布式**：Agent 可运行在不同进程或机器上；框架负责**序列化/反序列化消息**、**网络传输**（如 RPC/消息队列）、**路由与发现**（哪个 Agent 在哪个端点）。这样可水平扩展、隔离故障，并支持跨机多 Agent 协作。

#### 35 / 85. 通义 DeepResearch 的架构特点

面向“深度研究”类任务，通常结合：  
- **长文本**：对长文档做分块、摘要或检索，再注入；或使用长上下文模型。  
- **搜索增强**：多轮检索（先查概览再查细节）、多源检索（网页、论文、数据），结果去重与排序后作为上下文。  
- **Agent 架构**：规划（拆解研究问题）→ 检索/阅读 → 总结/综合 → 可能多轮迭代，形成“搜索-阅读-总结”闭环；部分步骤可并行以降低延迟。

#### 37 / 49. 流式输出 Agent：SSE 与后端执行配合

- **前端 SSE**：通过 EventSource 或 fetch + stream 消费服务端 SSE 流；服务端每产生一段内容（如 token、或一个 tool call 结果）就推一条 event，前端增量渲染。  
- **后端**：Agent 执行循环中，每得到一段生成结果就 flush 一次（如 token 级或句子级）；若遇 Function Call，可先推送“正在调用工具…”再推送工具结果与后续生成。  
- **Function Call 场景**：流式下模型可能先输出文本再输出 tool_calls；需在解析层支持“流式拼接完整 tool_calls 再执行”，或先流式输出文本部分，tool_calls 在完整接收后执行并再开一条流推送结果与后续回复。

### 5.9 多智能体

#### 38. 共享记忆与独立记忆；避免上下文爆炸

- **共享记忆**：所有 Agent 读写同一块记忆（如共享消息池、共享知识库），便于协作与一致事实；但若每个 Agent 都把整块记忆塞进自己的上下文，会迅速撑爆窗口。  
- **独立记忆**：每个 Agent 只保留与自己相关的历史或状态，上下文小，但需要显式传递信息才能协作。  
- **平衡**：共享“结构化摘要”或“当前任务相关片段”，而非原始全量历史；或采用**分层**：共享层只存关键结论与任务元数据，详细历史留在各 Agent 本地，按需检索注入。

#### 39 / 42 / 43. 协作与竞争；共识机制；避免无限循环

- **协作与竞争**：多 Agent 通常以协作完成共同目标为主；在需要“多方案择优”或“投票决策”时引入**竞争**（如多个 Agent 各自给出方案，再仲裁选优）。  
- **共识机制**：当多个 Agent 对同一决策（如代码修改）有不同意见时，可通过**投票、评分、或指定角色（如架构师一票否决）** 仲裁；也可引入“讨论轮”让 Agent 交换理由后再投票。  
- **无限循环**：多 Agent 互相回复、无人结束。对策：**轮数上限**；**终止条件**（如某 Agent 输出“最终结论”）；**检测重复**（连续 N 轮消息语义相似则强制结束或转人工）；**角色权限**（仅某角色可做最终拍板）。

#### 40. LLM Compiler 范式

**思路**：将复杂任务拆成多个**可并行的子任务**，像编译器做指令调度一样，先分析依赖，再在依赖允许的前提下**并行**执行多个 Agent 或多次 LLM 调用，最后合并结果。从而在保持正确性的前提下降低总延迟（从串行变并行）。

#### 41. 软件开发团队 Agent 的角色 Prompt

模拟产品、架构、开发等角色时，**关键属性**包括：  
- **角色身份与职责**（如“你是前端工程师，负责实现界面与交互”）。  
- **输入输出约定**（接受什么格式的 PRD/设计，输出什么格式的代码或文档）。  
- **约束与原则**（技术栈、规范、不做什么）。  
- **协作方式**（从谁那里拿什么、交付给谁）。  

通过为每个角色定制 System Prompt 并在多 Agent 流程中固定角色，可得到较稳定的分工行为。

#### 81. 多 Agent 通信方式

常见有：**共享消息队列/总线**（发布-订阅）、**直接请求-响应**（A 调 B 的 API）、**共享状态/黑板**（所有 Agent 读写同一状态对象）、**编排器调度**（中心节点把任务分给各 Agent 并汇总）。选择取决于延迟、解耦、可扩展性需求。

#### 83 / 84. LangChain 核心组件；为何 Graph 模式逐渐取代 Chain

**LangChain 核心组件**：Models（LLM/Embedding）、Prompts、Chains、Agents、Tools、Memory、Callbacks。**Chain 的局限**：线性或固定分支，循环与状态难表达。**Graph 取代**：LangGraph 以图+状态为一等公民，支持循环、多步、可中断与人机协同，更适合有状态 Agent。

### 5.10 工程、安全与评测

#### 45. 自我纠错 Prompt 设计

当工具执行报错时，让 Agent **自主修正**：在 System 中明确“若工具返回错误，请根据错误信息分析原因（如参数错误、权限不足），修改调用参数或换用其他工具后重试，并简要说明修正内容”。可配合 Few-shot 示例：一次错误返回 + 一次修正后的正确调用。

#### 46 / 66 / 67. 延迟与成本优化

- **除模型推理外的优化**：  
  - **网络 IO**：就近部署、连接复用、减少请求次数（合并工具调用、缓存）。  
  - **Prompt 预处理**：异步做检索/摘要，准备好再调 LLM；压缩 system/历史以减 token。  
  - **并行**：多步独立调用并行发；流式输出让首字更快（TTFT）。  

- **首字延迟 TTFT**：  
  - 模型：小模型或蒸馏模型、投机解码（Speculative Decoding）。  
  - 网络：CDN、边缘节点、长连接。  
  - Prompt：减少首包前的计算（如缓存 system、预取检索结果）。  

- **缓存**：  
  - 对相同或近似请求缓存 LLM 响应；对工具调用结果按参数缓存；对 embedding/检索结果缓存。  
  - 高频 Agent 服务可对“热点问题”做预生成或模板化回复。

#### 47 / 48 / 87. 评测集与可观测性

- **评测集**：除最终任务成功率外，可量化**中间步骤**：规划是否合理（与人工标注或规则对比）、工具选择是否正确、参数是否合法、步数是否在合理范围。指标可包括：任务完成率、平均步数、工具调用准确率、违规调用次数等。  
- **可观测性**：  
  - **日志**：每轮输入输出、工具调用与结果、错误堆栈。  
  - **追踪**：请求 ID 贯穿多步、多服务，便于排查延迟与失败。  
  - **指标**：QPS、延迟（含 TTFT、E2E）、错误率、Token 消耗、工具调用分布。  
  - **审计**：敏感操作留痕、Prompt 与响应抽样留存。

#### 50. Token 消耗评估与降低成本

- **评估**：按请求统计 input/output token，按模型单价算成本；可按用户/场景/模型维度聚合。  
- **降本**：压缩 prompt（摘要、去冗余）、缓存、用小模型或蒸馏模型、控制 max_tokens、对长上下文用检索替代全文注入。

#### 51 / 52. 权限、数据隔离与版本管理

- **权限与隔离**：多租户下按用户/组织隔离数据与执行环境；敏感工具按角色授权；审计日志区分租户。  
- **版本管理**：  
  - **Prompt 版本**：Prompt 模板纳入版本控制（Git），发布时带版本号，支持回滚与 A/B。  
  - **工具定义**：工具列表与 schema 版本化，兼容旧客户端时可保留多版本接口。

#### 59 / 80. JSON Mode 与 Grammar-constrained decoding（GBNF）

- **JSON Mode**：API 层要求模型“仅输出合法 JSON”，通过采样约束或后处理减少非 JSON 输出。  
- **Grammar-based decoding（如 GBNF）**：在解码时用**文法**限制下一个 token 的合法集合（如“当前在键名后只能接冒号”），使输出 100% 符合给定 JSON Schema 或其它格式，从根上避免格式错误，提高工具调用解析成功率。

### 5.11 场景题与综合

#### 68. 金融研报分析 Agent：长文档与闭环

**挑战**：上千页 PDF，直接塞进上下文会丢失或超窗。  
**思路**：  
- **长文档**：按章节/页切块，做摘要与向量索引；先根据“分析目标”检索相关块，再对关键块做精读或二次摘要；必要时用长上下文模型处理单章。  
- **闭环**：规划（分析目标 → 数据需求 → 图表需求）→ 检索/抽取数据 → 调用绘图工具 → 汇总结论；可多轮“发现新问题 → 再检索 → 再画图”。

#### 69. 若 Context Window 无限大，RAG 与向量检索还会存在吗？

**仍会存在**：  
- **成本与延迟**：无限窗口不等于“免费、无延迟”，全文注入成本与延迟仍高；检索只注入相关片段更经济。  
- **注意力质量**：超长序列中模型对中前部关注度仍可能下降；检索相当于把“重要部分”放到更有效位置。  
- **动态数据**：RAG 可接入实时、更新的知识，而“无限窗口”多为静态快照。  
**架构演变**：可能更多“长窗 + 检索”混合：长窗存当前任务相关，检索负责大规模、动态知识库。

#### 70 / 91 / 100. 瓶颈与形态（2026+）

- **瓶颈**：模型能力（推理、工具选择、长程依赖）、推理成本（算力与单价）、数据质量（指令数据、工具使用数据）都会制约落地；不同场景主要瓶颈不同（如 ToB 更关注成本与可控性）。  
- **形态**：  
  - **超级大脑**：单一大模型负责多数决策，能力强但成本与延迟集中。  
  - **协作蜂群**：多小模型/多 Agent 分工，延迟可分散、成本可优化，但协调与一致性更难。  
  - 从算力成本、延迟、可靠性三个维度权衡，会趋向“关键路径用强模型、边缘或并行用轻量模型/Agent”的混合形态。

#### 71. 多模态原生时代 Agent 感知的飞跃

**质的飞跃**：  
- 直接理解**图像/视频/音频**，不再依赖单独识别再转文本。  
- 可“看图说话”“听音执行”（如根据界面截图操作、根据语音指令订票），感知与行动更贴近人类。  
- 多模态检索与推理（如“找出与这张图最相似的段落”“根据图表回答问题”）成为原生能力。

#### 72 / 97. Skills 渐进加载与检索

**问题**：Skills 成百上千时全量注入导致上下文爆炸。  
**方案**：  
- **意图预测**：先用轻量模型或规则对当前 query 做意图/领域分类，只加载该领域相关 Skills。  
- **语义+元数据检索**：对 Skill 描述与元数据（标签、适用场景）做向量+关键词检索，只注入 Top-K 个最相关 Skill；可结合使用（先意图筛再检索）。

#### 75 / 57. 结构化提示与思维链注入

- **结构化提示**：用固定格式（如 XML 标签、Markdown 小节）组织 prompt，便于模型与解析器一致地识别“角色”“任务”“输出格式”；对 Agent 而言，结构化输出（如 JSON、明确区块）便于解析 tool_calls 与最终答案。  
- **思维链注入**：在 prompt 中要求“请逐步推理并写出中间步骤”，或提供 Few-shot 的 chain-of-thought 示例，引导模型先推理再给结论，减少直接瞎猜。

#### 78. 多 API 依赖（如先天气再航班）的协调

- **代码层**：编排器按 DAG 执行：先调天气 API，将结果作为变量传入下一节点再调航班 API；或由 Agent 规划“步骤 1 天气、步骤 2 航班”，执行器按序执行并把上步结果注入下步 prompt。  
- **Prompt 层**：在 System 中说明“若需要先获得 A 再请求 B，请先调用 A，根据返回结果再调用 B”，并确保工具返回被完整放入下一轮上下文。

#### 79. 工具使用能力评估指标

除任务成功率外，可包括：**工具选择准确率**（应调用的工具是否被调用）、**参数正确率**（必填与格式是否正确）、**多余调用率**（不应调而调）、**调用顺序合理性**（依赖是否满足）、**平均步数/Token**（效率）。

#### 88. 高并发 Agent 服务架构

典型组件：**负载均衡**（多实例）→ **API 网关**（鉴权、限流）→ **Agent 服务**（无状态，可水平扩展）→ **异步队列**（长任务入队，worker 消费）→ **缓存**（LLM 响应、工具结果、embedding）→ **存储**（会话/状态/审计）。关键：Agent 实例无状态，状态放 Redis/DB；长任务异步化避免占满连接。

#### 89. 深度研究 Agent 的搜索-阅读-总结闭环

设计要点：  
- **搜索**：多轮检索（先宽后窄）、多源（网页、论文、数据），查询改写与扩展。  
- **阅读**：检索结果排序与过滤，长文档分块+摘要或重点抽取，只把关键内容注入。  
- **总结**：按研究问题结构化综合，支持“发现缺口 → 再搜索 → 再阅读”的迭代，直到满足深度与覆盖度要求。

#### 90 / 99. Skills 是什么、构建经验；框架选型与评价指标

- **Skills**：Agent 的**可插拔能力单元**，通常包含描述（何时用、怎么用）、参数定义、实现（代码或 API）；Agent 按需加载 Skill 并调用。  
- **构建经验**：从高频场景出发；描述清晰带示例；参数 schema 完整；版本与文档维护。  
- **框架选型**：考虑生态、与现有栈集成、是否支持多模型/多 Agent、可观测性与运维成本。  
- **评价指标**：任务成功率、完成时间、用户满意度、成本（Token/调用）、安全事件数等，按业务取舍。

#### 93 / 94. 文件系统作为“外挂大脑”；文件检索策略

- **外挂大脑**：用文件系统（JSON/MD/代码文件）持久化计划、中间结果、总结、用户偏好；Agent 通过读写文件维持长期记忆与状态，相当于“外存”。  
- **文件检索**：  
  - **关键词**：简单、可解释，适合精确匹配。  
  - **向量检索**：按语义找相关文件或片段，适合模糊需求。  
  - **GraphRAG**：将文档建图（实体与关系），检索时结合图结构做多跳推理，适合强关联、多实体文档。

#### 96. Anthropic Prompt Caching

**思路**：对**不变的前缀**（如长 system prompt、静态知识库）做缓存，只对可变部分（如用户消息）重新计算；可显著降低重复 token 的成本与延迟。  
**工程上**：把“很少变化”的部分（system、工具描述、固定示例）放在前缀并设为可缓存；用户轮次、当前会话历史放在后缀。

#### 98. OpenClaw / Manus 类框架的改进点

此类框架多在**任务拆解**与**执行效率**上优化：更细粒度的子任务与依赖、更好的并行调度、与系统/IDE 的深度集成（如直接操作文件、终端）。相比 LangChain/LangGraph，往往更强调“可执行、可落地”的闭环与性能，而非通用链式组合。

#### 62. 多 Agent 死循环的检测与熔断

- **检测**：轮数上限、同一消息/语义重复出现、状态不再变化（如连续 N 轮无新工具调用）。  
- **熔断**：达到条件后强制结束当前分支、返回“未完成”或转人工；或触发“反思/仲裁”节点，由专门节点决定是否继续或终止。

### 6.1 微调方式

- **全量微调**：数据量、算力需求与适用场景。

- **成本**：显存/内存、量化、批处理与自动扩缩容。
- **可观测性**：日志、追踪、指标（延迟、错误率、token 消耗）。

## 七、评估与监控

### 7.1 效果评估

- **分类/抽取**：准确率、F1、混淆矩阵。
- **生成**：BLEU、ROUGE、BERTScore、人工评分、LLM-as-Judge。
- **RAG**：检索指标 + 生成质量 + 引用准确性。
- **Agent**：任务成功率、步骤效率、工具使用正确性。
- **A/B 测试**：流量切分、统计显著性、多指标权衡。

### 7.2 安全与合规

- **内容安全**：敏感词、违规检测、输出过滤与人工复核。
- **幻觉与事实性**：检索增强、引用、置信度与“不知道”的表述。
- **隐私**：输入脱敏、日志脱敏、数据留存策略。
- **合规**：数据属地、可解释性、审计与行业规范。
